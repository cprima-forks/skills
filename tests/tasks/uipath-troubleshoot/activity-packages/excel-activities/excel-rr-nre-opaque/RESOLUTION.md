# Final Resolution

---

**Outcome:** The `uip` CLI evidence confirms the playbook match
(read-range-null-reference) and rules out two of the five branches,
but it cannot positively identify which of branch 2 (structural
corruption) / branch 4 (unsupported OpenXML feature) / branch 5
(heavy formatting / scale) applies. Diagnosis requires manual
workbook inspection. The agent's correct action is to hand the
user a specific inspection checklist; the user's "Python script
generates the workbook nightly" clue justifies leaning toward
branch 2 (corruption from non-Excel writers) as the most likely
candidate, but still requires verification.

**What the CLI evidence does establish:**

- Failing job `cc333333-bbbb-cccc-dddd-eeeeffffaaaa` opened the
  workbook successfully (no `IOException`, no `FileNotFound`).
- The exception is `TargetInvocationException` wrapping a generic
  `NullReferenceException` from inside the activity, with no
  cell / sheet / range pointer in the stack trace.
- The fingerprint matches the read-range-null-reference playbook.

**What the CLI evidence RULES OUT:**

- **Branch 1 (sensitivity label):** no Purview / AIP / encryption
  evidence in job logs. The workbook opened without any label-
  enforcement message.
- **Branch 3 (broken named range):** workflow source uses
  `Range: ""` (empty / used-range read), not a named range. No
  defined-name lookup is happening.

**What the CLI evidence CANNOT determine:**

- **Branch 2 (structural corruption):** would need to open the
  workbook in Excel and observe the "Repaired" dialog or a
  recovery log.
- **Branch 4 (unsupported OpenXML feature):** would need to run
  Excel's Compatibility Checker against the workbook to enumerate
  features the OpenXML provider may not support, OR try the
  workflow under `Excel Application Scope` (COM) to see if it
  succeeds.
- **Branch 5 (heavy conditional formatting / scale):** would need
  to inspect file size, sheet count, populated cell counts, and
  count of conditional formatting rules.

**Branch 2 is the most likely candidate** based on the user's
clue that the workbook is generated by a nightly Python script:
non-Excel writers (especially `openpyxl`) routinely produce
structurally invalid XLSX files when their internal state is
inconsistent, when they don't emit all required OOXML parts, or
when they write partial output after an interrupted run.

**Recommended Fix (Resolution):**

### Step 1 — Inspection checklist for the user

Ask the user to perform the following checks on a workstation with
the workbook accessible (they don't need to be on MOCK-HOST; the
workbook itself is what's being inspected):

1. **Branch 2 verification (most likely given the Python writer):**
   - Open `C:\Robot\Data\sales-2026-05.xlsx` in Excel.
   - Watch for a "We found a problem with some content in
     [file]. Do you want us to try to recover as much as we
     can?" dialog. If yes, click Yes and review the recovery
     log. **The presence of this dialog confirms branch 2.**
   - If no dialog appears, the file is structurally valid; try
     branches 4 / 5.

2. **Branch 4 verification:**
   - Open the workbook in Excel.
   - **File -> Info -> Inspect Workbook -> Check for Issues
     -> Check Compatibility**.
   - Note any features flagged with "Significant loss of
     functionality" or "Minor loss of fidelity" -- these are
     candidates for features the OpenXML provider cannot
     handle. Common culprits: legacy macros, embedded OLE
     objects, certain chart types.

3. **Branch 5 verification:**
   - **File -> Info** -- note file size on disk.
   - Open each sheet, press `Ctrl+End` to jump to the last
     used cell, and note row / column count.
   - **Home -> Conditional Formatting -> Manage Rules** ->
     "Show formatting rules for: This Worksheet" -- count
     rules; > 100 is a candidate for branch 5.
   - File size > 100 MB or > 1M total populated cells supports
     branch 5.

4. **(Cross-check) Branch 3 once more:**
   - Press `Ctrl+F3` to open Name Manager. If any defined
     name shows `#REF!` in the `Refers to` column, the
     workflow might be using that named range indirectly --
     unlikely given the `Range: ""` empty read, but worth a
     quick check.

### Step 2 — Apply the per-branch fix

Once the user identifies the branch from inspection, apply:

- **Branch 2 (Repaired dialog appeared):** open + Save As in
  Excel to produce structurally clean output, then fix the
  Python writer upstream. Most likely fixes:
  - Upgrade `openpyxl` to the latest version (older versions
    had several known XLSX-structure bugs).
  - Switch to `xlsxwriter` (which writes cleaner XLSX) or to
    Excel COM via `pywin32` (most reliable for Excel-consumable
    output).
  - Ensure the Python script does NOT write partial output --
    write to a temp path and rename atomically only on success.
- **Branch 4 (Compatibility Checker flagged features):** remove
  the incompatible features OR set `ReadFormatting: True` on
  `Use Excel File` to force COM (requires Excel on the Robot
  host).
- **Branch 5 (large file / many rules):** reduce conditional
  formatting; compress pivot caches; split the workbook; or
  migrate to the cloud surface (`o365-activities` Graph reads
  scale better).

### Step 3 — Validation at boundary (prevention)

For any workbook generated by an upstream script, add a
boundary smoke test:

```vb
' Right after the workbook is produced (or at the start of
' this workflow), open it via Get Workbook Sheets in a Try
' Catch. If the smoke test throws, fail fast with a clear
' message naming the workbook path and asking for inspection.
Try
    Dim sheets = ExcelWorkbookScope.GetSheetNames()
    LogMessage($"Boundary check OK: {sheets.Count} sheets found")
Catch ex As Exception
    Throw New BusinessRuleException(
        $"Workbook smoke test failed for {workbookPath}: {ex.GetType().Name}: {ex.Message}. Open in Excel and check for Repaired dialog / Compatibility Checker warnings.")
End Try
```

This turns a generic NRE deep in the activity into a one-glance
diagnosis at job start, with a clear next-action prompt for the
operator.

### Long-term prevention

- Treat any externally-generated workbook as untrusted. Validate
  at the boundary where it enters the automation pipeline.
- Do not consume workbooks written by `openpyxl` (or other
  non-Excel libraries) directly without a sanitization step.
  The cheapest sanitization is opening + saving in Excel itself,
  which normalizes the XLSX structure.
- Document the workbook contract with the upstream owner: file
  format (current `.xlsx`), maximum size, no macros, no embedded
  OLE, generated by a specific tool with a known-good version.
