# Excel Read Range ‚Äî Opaque NRE / TIE, Needs Workbook Inspection

This scenario reproduces a Read Range failure where the workbook
opens successfully but parsing fails with a generic
`TargetInvocationException` wrapping an inner `NullReferenceException`.
The CLI evidence rules out the easier branches but cannot positively
identify which of the remaining (corruption / OpenXML feature / scale)
applies ‚Äî diagnosis requires manual workbook inspection. The job
ends with:

```
System.Reflection.TargetInvocationException: Exception has been thrown by the target of an invocation.
 ---> System.NullReferenceException: Object reference not set to an instance of an object.
```

## What this scenario uncovers

**Expected outcome:** The agent matches the playbook, recognizes
that:
- No sensitivity-label evidence in job logs (rules out branch 1).
- Workflow source uses an empty `Range` (not a named range), ruling
  out branch 3.
- The remaining candidates (branch 2 corruption / branch 4
  unsupported OpenXML feature / branch 5 heavy formatting) cannot
  be distinguished by CLI alone.

The agent then hands the user a workbook-inspection checklist (open
in Excel; look for "Repaired" dialog; Name Manager `Ctrl+F3` for
`#REF!`; Compatibility Checker; file size).

**Bonus signal:** the user mentions the workbook is "generated by
a Python script that runs nightly." This is a strong but not
conclusive lean toward branch 2 (structural corruption ‚Äî `openpyxl`
and similar libraries frequently produce structurally invalid
XLSX). The agent may name this as the most-likely candidate but
should still recommend verification before committing.

This maps to:
`skills/uipath-troubleshoot/references/activity-packages/excel-activities/playbooks/read-range-null-reference.md`
(collapsed branches 2/4/5 ‚Äî opaque NRE / TIE).
